# burpieland — Project Context

This document is a complete technical map of this repository: what it is, how every file relates to every other file, and how a request turns into a rendered page. It's written to be dropped into a Claude Project's knowledge base so a fresh conversation has full context without re-reading the repo.

## 1. What this is

A personal website/blog for **Avinash Reddy Arutla** (GitHub: `burpcat`), deployed to **`https://burpcat.github.io`** via GitHub Pages. It's a static site — no backend, no database, no real authentication, no analytics or tracking. Everything that looks dynamic (the "currently building X" status, the guestbook, dark mode) is either baked in at build time or handled client-side with `localStorage` / a third-party embed.

Site identity (from `src/_data/site.json`): displayed name `Avinash` (italic), tagline *"developer with strategy · occasional philosopher · poet"*, subtitle *"a personal site of 'Avinash Reddy Arutla' on the open web, circa 2026"*.

The repo name on disk is `burpieland`; the deployed repo/site is `burpcat.github.io`.

## 2. Tech stack

| Layer | Tool |
|---|---|
| Static site generator | [Eleventy (11ty)](https://www.11ty.dev/) v3 |
| Templating | Nunjucks (`.njk`) for layouts/pages, Markdown for post bodies |
| RSS/Atom | `@11ty/eleventy-plugin-rss` + a hand-written `feed.njk` |
| Styling | Plain CSS, custom properties (no framework, no Sass/PostCSS build step) |
| JS | Plain vanilla JS, no bundler, no framework |
| Comments/guestbook | [Giscus](https://giscus.app) (GitHub Discussions-backed embed) |
| Build-time data fetch | Node script hitting the public GitHub REST API |
| Hosting/CI | GitHub Actions → `gh-pages` branch → GitHub Pages |
| Fonts | Google Fonts (`Fraunces`, loaded via `<link>`), body text is monospace (`Courier New`) |

No test suite, no linter config, no TypeScript — this is intentionally a small, dependency-light project.

## 4. Build & dev workflow

`package.json` scripts:

- `npm start` → `eleventy --serve` — local dev server with live reload (localhost:8080).
- `npm run build` → `node scripts/fetch-github.js && eleventy` — **always** refreshes the GitHub status data before building. This is what CI runs.
- `npm run fetch` → just runs the GitHub fetch script, without a full rebuild (useful for refreshing `github.json` without waiting on 11ty).

There is no separate CSS/JS build step — `src/css/style.css` and `src/js/site.js` are passed straight through unmodified via `eleventyConfig.addPassthroughCopy(...)` (`.eleventy.js:9-10`).

## 5. Eleventy config (`.eleventy.js`) — the wiring hub

This file is the glue that makes every other piece function. Key pieces:

- **RSS plugin** (`addPlugin(rssPlugin)`) — provides filters used elsewhere, though the actual feed markup is hand-written in `feed.njk` rather than using the plugin's built-in feed generator.
- **`posts` collection** — `getFilteredByGlob("src/posts/*.md")`, sorted newest-first by date. This collection (`collections.posts`) is what the homepage, `/blog/`, and `/feed.xml` all iterate over.
- **Custom filters**, each used in multiple templates:
  - `limit(arr, n)` — truncates an array; homepage uses `collections.posts | limit(5)` to show only the 5 most recent posts.
  - `dotDate(dateObj)` — formats a date as `YYYY·MM·DD` (the site's stylized date format, used on post listings and the post page itself).
  - `readingTime(content)` — strips HTML tags, counts words, divides by 220 wpm, rounds to whole minutes (min 1). Applied to `post.templateContent` (the rendered HTML) wherever a reading-time badge appears.
  - `isRecent(date, days=14)` — true if a post's date is within the last N days. Drives the `NEW` badge on post titles.
  - `isoDate(dateObj)` — ISO-8601 formatting, used for `<time datetime="">` attributes and the Atom feed's `<updated>`/`<id>` fields.

## 6. Data files (`src/_data/`) — the content that isn't code

Eleventy exposes every file in `_data/` as a global template variable named after the file (minus extension). So `site.json` → `{{ site.* }}`, `experience.json` → `{{ experience.* }}`, etc.

- **`site.json`** — the single source of truth for site identity: `name` ("burpcat"), `nameDisplay` (`<em>Avinash</em>`, rendered with `| safe` so the italics work), `tagline`, `subTagline`, `url`, `language`, `author.{name,email}`, `githubUsername` ("burpcat"), and `manualStatus` (fallback text for the utility strip when GitHub data isn't available). Consumed in `base.njk` (title, meta description, header text) and in `about.njk` (mailto link, GitHub link) and `feed.njk`/`feed.xml` (feed metadata).

- **`experience.json`** — two arrays, `work` and `projects`, rendered on the homepage's left column (`index.njk:8-36`). Currently populated with 4 jobs (SmartPac, Mass General Brigham, Shiryam Technologies, Aapoon) and 4 projects (MadCP, shrutz, Softrifle Market Analytics, MedPix 2.0 Pythia). If either array is empty, the template shows an italic "empty state" message instead — this is a deliberate design decision (see `CONTENT.md`, which explicitly documents this project as a reusable starter template for other people to personalize).

- **`blogroll.json`** — a `feeds` array (name/url/feed/note) rendered at the bottom of the homepage under "— WHAT I'M READING" (`index.njk:74-88`). The whole section is conditionally rendered — it disappears entirely if the array is empty. Currently lists Paul Graham, Gwern Branwen, Lil'Log, and The Pragmatic Engineer.

- **`github.json`** — **not hand-edited**. This is regenerated on every build by `scripts/fetch-github.js` and is what powers the "currently building" line in the utility strip at the top of every page (`base.njk:33-42`). Current committed snapshot shows `repoName: "shrutz"`, pulled from the most recent public `PushEvent` on `github.com/burpcat`.

## 7. `scripts/fetch-github.js` — the only "dynamic" build step

Runs before every `eleventy` build (wired into `npm run build`). Logic:

1. Reads `githubUsername` from `site.json`.
2. If missing or still the placeholder `"yourusername"`, writes `{configured: false}` and exits — the site still builds fine, just shows the `manualStatus` fallback text.
3. Otherwise calls `GET https://api.github.com/users/{username}/events/public` (unauthenticated — no token, so subject to GitHub's anonymous rate limit) and finds the most recent `PushEvent`.
4. If found, also fetches `GET /repos/{owner}/{repo}` for the repo's description, then writes `{configured, available, repoName, repoUrl, description, commitMessage, lastPush}` to `src/_data/github.json`.
5. Any failure at any step degrades gracefully — it writes `{configured: true, available: false}` rather than throwing, so a GitHub API outage or rate-limit never breaks the deploy.

This is why the GitHub Actions workflow reruns the build every 6 hours on a cron schedule even without new commits — it's the mechanism that keeps the "currently building" status current.

## 8. Templates and layout system

**`src/_includes/base.njk`** is the single shared shell every page renders through (`layout: base.njk` in front matter). It contains:

- `<head>`: dynamic `<title>` (`{{ title }} · {{ site.name }}`), meta description falling back to `site.tagline`, RSS `<link>`, Google Fonts link, stylesheet link, and an **anti-flash theme script** — a synchronous inline `<script>` that reads `localStorage.theme` (or OS preference via `prefers-color-scheme`) and sets `data-theme` on `<html>` *before first paint*, avoiding a light/dark flash.
- **Utility strip**: left side shows either the live GitHub status (if `github.configured && github.available`) or `site.manualStatus`; right side has the theme toggle button, the "calm mode" toggle button, and a static "★ under construction ★" sticker.
- **Header**: site name/tagline/sub-tagline from `site.json`, then a nav row with links to `/`, `/blog/`, `/about/`, `/guestbook/`, `/feeds/`, an RSS icon link to `/feed.xml`, and non-functional `log in` / `sign up` buttons that open a placeholder modal (see §11 — this is intentionally not wired to any real backend).
- **Hero banner**: only rendered when `page.url == "/"` (homepage only) — a pure-CSS parallax mountain/stars/orb illustration, no images, built entirely from `clip-path` polygons and gradients that respond to the `data-theme` attribute.
- `{{ content | safe }}` — this is where each page's own template content is injected.
- **Footer**: "coded on aegon · no analytics OR tracking" plus links to RSS and guestbook. ("Aegon" appears to be the name the author gives their dev machine — it's also referenced as the sign-off in the first blog post: "sent from Aegon".)
- A hidden **auth modal** (`#authModal`) — shared by both login/signup buttons, content swapped by `openModal(mode)` in JS. Has real-looking email/password inputs but no submit handler that does anything — it's UI-only chrome, not a real auth flow.

**`src/_includes/post.njk`** wraps individual blog posts: date + reading time row, title, the markdown-rendered body (`{{ content | safe }}`), and a "← back to all writing" link.

## 9. Pages

- **`index.njk`** (`/`) — two-column grid (`main-grid`): left column is Experience + Projects from `experience.json`; right column is the 5 most recent posts (title, date, reading time, excerpt, NEW badge if `isRecent`) plus a guestbook teaser box, with a "all writing →" link if there are more than 5 posts. Below the grid, the blogroll section (conditionally rendered).
- **`about.njk`** (`/about/`) — static prose, hand-written personal bio (voice: reflective/philosophical, references first-principles thinking, self-taught background, "AI is the new steam engine" framing). Ends with a mailto link (`site.author.email`) and GitHub link (`site.githubUsername`), both pulled from data rather than hardcoded.
- **`blog/index.njk`** (`/blog/`) — same post-card markup as the homepage's writing section, but lists *all* posts, no limit. Shows a "how to write your first post" code sample if the collection is empty.
- **`guestbook.njk`** (`/guestbook/`) — short intro paragraph, then a live **Giscus** `<script>` embed pointed at `burpcat/burpcat.github.io` (repo ID `R_kgDOTEg8ug`), category "Guestbook" (category ID `DIC_kwDOTEg8us4C_6kx`), mapped to the specific term `"guestbook"`. This means every comment left here becomes a GitHub Discussion post in that repo — no custom backend needed, moderation happens via GitHub's own tools.
- **`feed.njk`** (`/feed.xml`) — hand-written Atom feed (not generated by the RSS plugin's own feed builder, though the plugin is installed and its filters could be used). Iterates `collections.posts`, uses `isoDate` for timestamps and wraps each post body in `CDATA`. Marked `eleventyExcludeFromCollections: true` so it isn't treated as a page needing the `base.njk` layout.
- **`feeds.njk`** (`/feeds/`) — a plain-English explainer of what RSS is and how to subscribe, links to a few reader apps (NetNewsWire, Feedly, Inoreader, Reeder), and points at the site's own `/feed.xml`. Contains a small personal aside crediting "anji" for introducing the author to RSS.

## 10. Blog posts (`src/posts/`)

`posts.json` is an Eleventy **directory data file** — its contents (`layout: post.njk`, `tags: ["posts"]`, `permalink: "/blog/{{ page.fileSlug }}/"`) apply as default front matter to every file in `src/posts/`. Individual posts only need `title`, `date`, `excerpt` in their own front matter; everything else (layout, URL structure, collection membership) is inherited.

Filename convention: `YYYY-MM-DD-slug.md` → URL `/blog/YYYY-MM-DD-slug/` (the date prefix is for sorting/organization; the actual page date comes from the `date:` front matter field, and the URL slug comes from the filename via `page.fileSlug`).

Currently one post exists: **`2026-06-26-the-pattern-repeats.md`** — "The pattern repeats," an essay-style opening post arguing that today's expensive AI/agent compute is analogous to the historically-scarce, historically-rationed early computing era ("computer" as a job title, mainframe time rationing) — framing high current AI costs as a temporary floor rather than a permanent ceiling. Signed "sent from Aegon."

## 11. Styling (`src/css/style.css`, 770 lines)

Single stylesheet, no preprocessor. Structure:

- **Theming via CSS custom properties**: `:root, [data-theme="light"]` defines the light palette (warm parchment background `#E8E2D4`, dark ink text, forest-green accent `#2D5448`); `[data-theme="dark"]` overrides the same variable names with a navy/warm-accent dark palette. Every themed rule references `var(--x)` rather than literal colors, so the entire site re-themes by flipping one attribute on `<html>`.
- **Reduced-motion support**: a `prefers-reduced-motion: reduce` media query collapses all animations/transitions to near-zero duration.
- **"Calm mode"**: a `body.calm-mode` class (toggled by JS, see §12) turns off the pulsing status dot, the NEW badge pulse animation, and the ember particle animation, and hides the "under construction" sticker — an accessibility/low-stimulation affordance layered independently from the reduced-motion media query.
- **Hero banner** is built entirely from layered `<div>`s with CSS `clip-path` polygons (`.hero-far`, `.hero-near`, `.hero-trees`) plus a radial-glow "orb" (`.hero-orb`) and floating "ember" particles that only animate in dark mode.
- **Typography**: display/heading font is `Fraunces` (serif, loaded from Google Fonts, italic variant used for emphasis and taglines); body font is monospace (`Courier New`) — a deliberate old-web/terminal aesthetic contrasted with the elegant serif for headings.
- **Layout**: `.main-grid` is a 2-column CSS grid on the homepage, collapsing to 1 column (with post list ordered above experience) under `max-width: 860px`. Two responsive breakpoints total (860px, 560px).
- All interactive elements maintain a `44px` minimum tap target (`--tap-min`) for touch accessibility, and there's a visible `.skip-link` for keyboard/screen-reader users to jump past the header.

## 12. Client JS (`src/js/site.js`, 77 lines)

Three independent IIFEs/handlers, no dependencies:

1. **Auth modal** — `openModal(mode)` / `closeModal()` swap title/description text for `login` vs `signup` and toggle an `.open` class on `#authModal`; Escape key closes it. Purely cosmetic — there is no form submission logic, no backend, no session handling anywhere in the codebase.
2. **Theme switch** — reads the current `data-theme` (already set by the inline anti-flash script in `<head>`), syncs the toggle icon (☾/☀), and on click flips the attribute and persists the choice to `localStorage['theme']`. Also listens for OS-level `prefers-color-scheme` changes and follows them live *only if* the user hasn't made an explicit choice yet (i.e., explicit choice always wins over OS setting).
3. **Calm mode** — persists to `localStorage['calm-mode']`, defaults to "on" automatically if the OS has `prefers-reduced-motion: reduce` set, and toggles the `body.calm-mode` class described in §11.

## 13. Deployment (`.github/workflows/deploy.yml`)

GitHub Actions workflow, triggers on:
- push to `master`,
- a cron schedule (`0 */6 * * *` — every 6 hours), specifically to keep the GitHub "currently building" status fresh even with no new commits,
- manual dispatch.

Steps: checkout → Node 24 setup (with npm cache) → `npm ci` → `npm run build` (which runs `fetch-github.js` then `eleventy`) → deploy `_site/` to the `gh-pages` branch via `peaceiris/actions-gh-pages@v4`, using the default `GITHUB_TOKEN`. GitHub Pages is configured (per `CONTENT.md`'s setup instructions) to serve from that `gh-pages` branch. `_site/` itself is gitignored in `master` — it only exists as a build artifact, either locally or transiently in CI before being pushed to `gh-pages`.

## 15. `CONTENT.md` — what it actually is

Worth understanding separately from this file: `CONTENT.md` (repo root, currently untracked in git) is a **personalization checklist**, written as if handing this codebase to someone else as a reusable starter template ("Make this site yours"). It walks through editing `site.json`, `experience.json`, `blogroll.json`, writing the about page, adding a first post, wiring up Giscus, and deploying — implying this project's structure was either built as, or is being treated as, a general-purpose "personal site starter kit" pattern, with Avinash's own site as the reference instance of it.

## 16. How it all syncs together — end-to-end data flow

1. **Build starts** (`npm run build`, or CI) → `fetch-github.js` runs first, hits the GitHub API, writes `src/_data/github.json`.
2. **Eleventy runs** → reads `.eleventy.js` config → loads every `_data/*.json` file as a global variable → builds the `posts` collection from `src/posts/*.md` (each post's front matter merged with the `posts.json` directory defaults) → renders every `.njk` page through `base.njk` (unless excluded, like `feed.xml`).
3. **Every page** pulls `site.*` for identity/branding, `github.*` for the live status line, and (on relevant pages) `experience.*`, `blogroll.*`, or `collections.posts` for content.
4. **Static assets** (`css/`, `js/`) are passed through unchanged into `_site/`.
5. **Output** (`_site/`) is what CI pushes to `gh-pages`, which GitHub Pages serves at `burpcat.github.io`.
6. **At runtime in the browser**, the only "logic" is: the anti-flash inline script picks the theme before paint → `site.js` keeps the toggle UI in sync and persists choices to `localStorage` → Giscus's own script (unrelated to the rest of the build) renders the comment widget by talking directly to GitHub's API from the client.

There is no server-side runtime at all — once `_site/` is generated, the "app" is fully static except for the two third-party client-side scripts (Google Fonts loader, Giscus).

## 17. Things a future assistant should know before touching this repo

- The `log in` / `sign up` buttons and modal are **decorative** — do not assume there's an auth system to extend; there isn't one anywhere in the code.
- `src/_data/github.json` is a **build artifact that happens to be committed** — don't hand-edit it as if it were static config; it gets overwritten on every build.
- Adding a blog post is just dropping a correctly-named `.md` file with `title`/`date`/`excerpt` front matter into `src/posts/` — no other file needs to change.
- The site deliberately has no analytics, counters, or tracking — don't add any without explicit instruction, it goes against a stated design principle in the footer copy and `CONTENT.md`.
- Theme/calm-mode state lives only in `localStorage`, per-browser — there's no user account system to sync it.
- The real remote is `https://github.com/burpcat/burpcat.github.io.git`, deploying via the `gh-pages` branch, not `master` directly.
